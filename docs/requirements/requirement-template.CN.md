# 需求管理文档模板

状态：模板
最近更新：2026-06-04

## 使用方式

本模板用于创建 `docs/requirements` 下的单个需求管理文档。它提供轻量结构，帮助记录需求目标、范围、状态、已完成子目标、剩余子目标和相关知识库链接。

模板不是固定格式。编码 AI 可以根据需求规模增删章节、调整标题或合并字段；重点是保留需求级进度，不把临时任务计划、文件级 TODO 或详细实现设计放进需求管理文档。

本模板面向涉及项目非测试代码改动的需求。仅文档、仅测试、工作流调整或仓库 housekeeping 通常不需要创建需求管理文档。

## 建议模板

```md
# REQ-YYYY-MM-short-name

状态：Discussing
创建日期：YYYY-MM-DD
最近更新：YYYY-MM-DD

## 背景

简要说明需求出现的原因、用户问题或产品上下文。

## 目标

描述需求希望达成的结果。这里关注用户价值或系统能力，不展开具体实现方案。

## 范围

- 需求包含的能力或文档调整。
- 需求影响的模块、语言或 provider。

## 非范围

- 明确暂不处理的内容。
- 避免编码 AI 在实现时误扩展需求边界。

## 当前进度

### 已完成子目标

- 记录已经落地的需求级结果。

### 剩余子目标

- 记录仍未完成的需求级结果。

## 相关知识库文档

- 设计文档：
- 实现文档：
- 用户文档：
- 第三方资料：

## 开放问题

- 尚未确认的设计、范围或验收问题。
```

## 维护建议

- 需求状态和粗粒度进度放在需求文档与 [requirements/STATUS.md](STATUS.md) 中。
- 系统设计内容放在 `docs/design`。
- 实现设计和实现基线放在 `docs/impls`。
- 用户可见行为和 API 用法放在 `docs/user`。
- 设计或实现文档可以链接回引入它的需求，但不需要维护该需求的当前状态。
